# FB広告結果とデリバリー分析（アカウント0件の要因と改善方針）

## 1. 現状サマリ

| 項目 | 英語（リール動画） | 日本語（静止画） |
|------|-------------------|------------------|
| 予算 | 約250円/日 × 2週間 | 約250円/日 × 2週間 |
| クリック単価 | **約1円**（安い） | **約23円**（高い） |
| サイト訪問 | あり | あり |
| **アカウント作成** | **0人** | **0人** |

**結論**: クリックは取れているが、着地後の「価値体験」がなく、登録に至っていない。**デリバリーのボトルネックは「獲得」ではなく「起動」（初回体験）にある。**

---

## 2. デリバリー観点での診断

### 2.1 何が起きているか

- 広告では「無料で体験」「Google連携で始める」と訴求している。
- ユーザーは「無料で試せる」と期待してサイトに来る。
- 着地後は **「Googleアカウントで無料開始」のCTAのみ**。  
  → **価値を一度も体験する前に「登録」を求めている**。
- その結果、「このサービスで何が得られるか」を信じる前に離脱している。

デリバリーの言葉で言うと:

- **獲得**: 「これ、自分の問題だ」と思わせるまでが獲得。広告のクリックは「興味」であって、まだ「自分の問題」とまでは思っていない可能性が高い。
- **起動**: 「やればすぐ得する」と思わせる段階。**現状、ログインしないと何も得られない = Time to Value が「登録後」にあり、起動が成立していない。**

つまり、**プロダクトの価値はあるが、「価値に出会う導線」が登録の前になく、デリバリー設計が欠けている**状態です。

### 2.2 設計ドキュメントと実装のギャップ

`docs/user-journey.md` では次のように設計されています。

- **ステップ1（完全無料・未ログイン）**:  
  URL入力 → GSC不要で現在順位＋改善のヒント（1つ）を表示 → 「続きはアカウント作成で」
- **ステップ2**: アカウント作成 → 完全な分析結果・無料枠

しかし **現在のLP** では:

- 未ログイン時は **Hero（「Googleアカウントで無料開始」）→ Features → How it works → Pricing → FAQ** のみ。
- **URLを入力して「改善のヒント」を一度でも見る**体験は、**ログイン後の AuthenticatedContent にしか存在しない**。

つまり、「登録前に驚きを与える」という設計は **ドキュメント上にはあるが、プロダクトには実装されていない**。  
このギャップが、**アカウント0件の最大の構造的要因**です。

## 3. デリバリーファネルで整理する

### 獲得 → 起動 → 習慣 → 拡散 → 収益

各段で「ユーザーが何を信じる必要があるか」を言語化すると:

| 段階 | 信じてもらうこと | 現状 |
|------|------------------|------|
| **獲得** | 「これ、自分の問題だ」 | 広告でクリックは取れているが、着地後の一言・体験が弱く、自分事化しづらい |
| **起動** | 「やればすぐ得する」 | **未ログインでは何も得られない** → 起動が成立していない |
| 習慣 | 「これがないと戻れない」 | 起動していないため未到達 |
| 拡散 | 「人に教えると得になる」 | 未到達 |
| 収益 | 「払う方が合理的」 | 未到達 |

**ボトルネック**: 獲得の「言葉」と、**起動の「初回体験の編集」**（登録前に一度価値を見せること）。

記事で触れられていたポイントの通り:

- **起動が弱い理由は、オンボーディングの不足**（ここでは「登録前の価値体験」の不足）。
- **Time to Value が遅い**（価値体験が「ログイン後」にある）と、その前に離脱する。

## 4. ターゲット・売り方・「刺さっていないか」の整理

### 4.1 サービスが刺さっていないか？

- **クリック単価**: 英語1円・日本語23円で、少なくとも「興味」はある層がクリックしている。
- **0登録** は、**「興味はあるが、信じる理由が足りない」「体験する前に登録を求められて離脱」**の可能性が高い。
- つまり、**「刺さっていない」というより、「刺さる前に離脱している」**と考えるのが自然です。
- ただし、**本当に「SEO記事の順位メンテ」が自分の問題だと思っている人**に届いているかは、ターゲット設定次第（後述）。

### 4.2 ターゲット

- **「誰の、どんな課題」**をはっきりさせる必要がある。
  - 例: 「自社ブログ・メディアでSEO記事を書いているが、メンテまで手が回らず、気づいたら順位が下がっていた経験がある人」
- 広告のオーディエンスが **「興味はあるが、今まさにその課題を抱えていない人」** だと、クリックはしても「自分の問題」と思わず、登録まで至りにくい。
- **日本語23円** は、**ターゲットが広い／競合入札が高い／クリエイティブの訴求が弱い**のいずれかの可能性。  
  → 日本語は「超特定」（例: ブログ運営者・コンテンツ担当・SEO担当）に絞り、**「順位が下がった／メンテが大変」という一言**で自分事にさせる必要がある。

### 4.3 売り方（届け方）

- **現状**: 「無料で体験」と訴求しているが、着地後は「登録してから体験」になっている。
- **あるべき形**:  
  - **① 既存の流れに乗る**: SEO記事・GSC・コンテンツメンテで既に困っている人がいる場所（検索・ブログ・コミュニティ）にコンテンツや連携で乗る。  
  - **② 最初の成果物を自動で出る**: 未ログインで「URL入力 → 順位＋改善のヒント1つ」まで見せる。  
  - **③ 一人が連れてくる構造**: 分析結果や改善案を共有したときにロゴ・リンクが付くなど（中長期）。
- 広告は「チャネルの一つ」であり、**プロダクト側で「登録前に価値を見せる」形になっていないと、広告費は燃えやすい**、という状態です。

## 5. 具体的な改善アクション（優先度順）

### 5.1 最優先: 登録前の「一度だけ価値を見せる」体験を実装する

- **やること**: `user-journey.md` のステップ1を実装する。
  - 未ログインでも **記事URL** と **調べたいキーワード**（1つ）を入力できるフォームをLP上に配置。  
    （キーワードは「この記事で狙っているメインキーワード」をユーザーが入力。GSCがないと「どのキーワードで見るか」をこちらでは分からないため。）
  - バックエンド: **Serper API** でそのキーワードを検索 → 結果一覧で該当URLの位置を「順位」として取得 → 改善のヒントを1つだけ返す（GSC連携不要）。詳細は `docs/unauthenticated-rank-check.md` を参照。
  - 表示: 「このキーワードで、あなたの記事は現在○位です」「上位記事に比べて、ここが足りません: ○○」など。
  - その後に「続き（詳細な改善案・リライト原稿）を見る → 無料アカウント作成」と誘導。
- **効果**: Time to Value を「登録前」に前倒しし、**「やればすぐ得する」を信じてもらってから**登録してもらう。
- これが、デリバリーで言う **「初回体験の編集」** と **「最初の成果物が自動で出る」** に相当する。

### 5.2 ランディングの「一言」と証拠の整理

- **誰のどんな課題か** を1行で決める。  
  例: 「SEO記事を書いたあと、メンテまで手が回らず、気づいたら順位が下がっていた人」
- その課題に対して **刺さる一言を10個** 書き、広告・LPの見出し・サブコピーに使う。  
  例: 「順位が下がってからじゃ遅い。今日から、AIが差分を教えます。」
- **証拠**: デモ動画（既存）、「未ログインで試せる」体験（上記）、可能なら数値・事例を追加。
- Hero の流れを **「まずURLを入れて結果を1つ見る → 続きは無料登録」** に寄せる。

### 5.3 広告のターゲット・クリエイティブ

- **日本語**:  
  - オーディエンスを **ブログ／メディア運営・SEO・コンテンツ担当** などに絞る。  
  - 「順位が下がった」「記事メンテが大変」など、**問題を自分事にさせるコピー**を優先。  
  - 可能ならリール動画も試し、単価改善を図る。
- **英語**:  
  - 単価は既に低いので、**着地後の体験（登録前の1回体験）** を実装したうえで、同じ予算で再度計測。
- いずれも **「無料で今すぐ1本だけ試せる」** と訴求し、LPと体験を一致させる。

### 5.4 チャネルを広告だけにしない

- **コンテンツ起点**: 「順位下落 原因」「SEO 記事 メンテ」「GSC 活用」など、既に検索している人向けの記事を増やし、ブログ／SEOから流入を作る。
- **ワークフロー寄生**: GSCを既に使っている人向けに、「GSCと連携するだけで」を前面に出す。既存の作業導線に入る形で価値を伝える。

## 6. 指標の置き方（起業初期）

- **まず見るべき**:  
  - **Activation** = 価値を体験した人の割合。  
    → 「未ログインで1回分析結果を見た人」を分母に、「そのうちアカウント作成した人」の割合を追う。  
  - **Time to Value** = 初回の価値体験までの時間。  
    → 「URL入力 → 結果表示」を30秒以内にするなど、短くする。
- そのうえで、**継続が起きる理由**（定期チェック・通知・複数記事）を設計し、収益化はその後でよい。
- 支払い理由（時間短縮・精度・社内展開・監査など）は、有料化の前に言語化しておく。

---

## 7. まとめ

- **アカウント0件の主因**: 「無料で体験できる」と広告で言っているのに、**実際は登録しないと何も体験できない**。デリバリーの「起動」が設計されておらず、Time to Value が遅い。
- **設計と実装のギャップ**: `user-journey.md` の「未ログインでURL入力→改善のヒント」は **未実装**。ここを実装することが最優先。
- **ターゲット・売り方**: サービスが刺さっていないというより、**届ける形（登録前の価値体験）が欠けている**。ターゲットを「SEO記事メンテで困っている人」に絞り、一言と証拠を整え、広告とLPを一致させる。
- **デリバリーを最初に考える**とは、今回の場合は「**どの一言で、どの体験をさせ、その結果どの指標が動けば勝ちか**」を決め、そのために **LPと初回体験の仕様を逆算する**こと。  
  → その意味で、**「登録前に1回だけ価値を見せる」** 実装が、デリバリー設計の第一歩です。

次の実装タスクは、**未ログイン時の「URL入力 → 順位＋改善のヒント1つ表示 → 続きはアカウント作成」** フローの具体化と、LPへの組み込みです。
